## Summary

Resolved the long-running `unitAI-oyyt2` epic — "SQLite-first observability:
remove runtime dependence on `.specialists/jobs/`" — from a P0 handoff brief
through final integration. Discovered mid-session that the original `unitAI-ppkdg`
write-removal slice was too coarse for a single bead, and split it deliberately
twice (first into pre-migration / write-gate / post-cleanup, then split the
pre-migration slice itself into `fln4q-A` for 13 readers with existing DB parity
and `fln4q-B` for 6 file-only readers without DB equivalents) after an overthinker
critique flagged crash-recovery DB parity as the missing precondition. Shipped 8
worktree branches each with its own contract+executor+reviewer chain, with one
branch (`fln4q-B2` watchdog) requiring a second overthinker pass to settle the
architectural choice between embedding `bun:sqlite` in the detached child vs an
IPC channel back to the parent (option-a won — IPC defeats the watchdog's purpose
of detecting parent death). Merged all 8 branches into master with rebase chain
sequencing (4ci32 → m5i0e → 1ypy2 → t8o6o → fln4q-A → fln4q-B2 → ppkdg → jjp7w),
hand-resolved supervisor.ts conflicts where `shouldWriteJobFiles()` and
`isJobFileOutputEnabled` had collided across the cluster, then validated end-to-end
against the freshly-built binary — caught and fixed a merge artifact in 4
`withSqliteOperation` callbacks where `persisted === undefined` was firing
spuriously because the inner callbacks returned `void` (the check was meant to
detect a null SQLite client but couldn't disambiguate). E2E confirms both modes
work: default-off yields DB rows + zero file artifacts, file-on
(`SPECIALISTS_JOB_FILE_OUTPUT=on`) yields DB + status.json + events.jsonl, and
runtime mode is observable via `sp status --json`.

Two cross-cutting discoveries shaped the session: (1) **specialists routinely
leave work staged-but-uncommitted** even after passing tests/lint/build —
recurring across 6 of the 8 chains, requiring follow-up commit-only dispatches or
direct git commits — and (2) **specialists with `gitnexus-required` template
don't have GitNexus MCP tools loaded in the harness**, so they fall back to
grep/Read instead, leaving a traceability gap that reviewers keep flagging
(filed as `unitAI-c6he0`). Also corrected an early misconception about steering
specialists by raw token count — the `using-specialists-v2` 65-80% threshold is
context-window percentage, not absolute tokens; gpt-5.3-codex with ~200k window
is healthy at 90k.
